Sound is one of the most natural forms of data in the world, and capturing it digitally is at the heart of countless modern devices — from voice assistants to conference phones. In this project, we'll connect two I2S MEMS microphones to a Lattice iCE40UP5K FPGA, implement the I2S protocol entirely in Verilog, and stream the captured 16-bit stereo audio to a laptop over UART where a Python script saves it as a standard WAV file.

As you build this project you'll learn how to:

- Understand the I2S serial audio protocol
- Generate clocks and timing signals in Verilog
- Shift serial data into parallel registers
- Build a UART transmitter from scratch
- Debounce a push button in hardware
- Use the iCE40UP5K's built-in oscillator and RGB LED driver primitives
- Capture serial data with Python and write WAV files

## Requirements

- Lattice iCE40UP5K Breakout EVN (B-EVN) board
- 2x Adafruit SPH0645LM4H I2S MEMS Microphone breakout boards
- Jumper wires and a small breadboard
- USB cable (for programming the FPGA and UART communication)
- Python 3 with `pyserial` installed on your laptop

## The I2S Protocol

Before writing any Verilog, it's worth spending time understanding the I2S (Inter-IC Sound) protocol. I2S was developed by Philips Semiconductor (now NXP) in 1986 and remains the dominant standard for transferring digital audio between chips.

### Signals

I2S uses three wires:

- **SCK** (Serial ClocK) — the bit clock, generated by the controller. Each rising edge marks one data bit.
- **WS** (Word Select) — sometimes called LRCK (Left/Right CLocK). Toggles at the sample rate to indicate which channel's data is on the wire.
- **SD** (Serial Data) — the actual audio, MSB first, one bit per SCK cycle.

Figure 1 shows the basic system configurations and timing diagram from the I2S specification.

![I2S system configurations and timing diagram](/images/2026/06/i2s-timing.png)
*Figure 1: I2S system configurations and basic interface timing (from Philips I2S spec)*

### Timing

The most important timing detail is shown in the waveforms in Figure 1. Notice that the MSB of the serial data appears **one SCK cycle after the WS transition** — not on the transition itself. This one-cycle delay is a deliberate part of the I2S spec. It exists so that a receiver can detect the WS edge and still have a full clock cycle to prepare before the first data bit arrives. When implementing the receiver in Verilog we must skip this first slot and start capturing from bit index 1.

### Channel Mapping

When **WS is low**, the right channel data is being transmitted. When **WS is high**, the left channel is on the wire. At 16 bits per channel with 32 SCK cycles per WS period (16 data bits + 16 padding bits), the sample rate is simply:

```
Sample Rate = SCK frequency / 64
```

### The SPH0645 MEMS Microphone

The Adafruit SPH0645LM4H breakout houses a MEMS microphone with a built-in sigma-delta ADC that outputs 18-bit audio over I2S. It is powered at 3.3V — a perfect match for the iCE40UP5K's GPIO voltage — so no level shifting is required.

The breakout has a **SELECT** pin that determines which WS phase the mic drives its data on:

- **SELECT = GND** → the mic outputs data during **WS = 1** (left channel)
- **SELECT = VDD** → the mic outputs data during **WS = 0** (right channel)

When it is not its turn, the mic's SD output is in high-impedance (tri-state), so you can wire both mics' SD pins together, or use separate FPGA input pins as we do here. We capture the top 16 of the 18 available bits, giving us 16-bit audio at 15625 Hz.

### Wiring the Microphones

Connect the two SPH0645 breakouts as follows. Both mics share the SCK and WS signals from the FPGA — only the DATA and SELECT pins differ.

| SPH0645 Pin | Left Mic | Right Mic |
|-------------|----------|-----------|
| 3V          | 3V3 on FPGA board | same |
| GND         | GND | GND |
| BCLK (SCK)  | i2s_sck (FPGA pin 19) | same |
| LRCLK (WS)  | i2s_ws  (FPGA pin 18) | same |
| DATA        | i2s_sd_left (FPGA pin 20) | i2s_sd_right (FPGA pin 21) |
| SELECT      | GND (left channel) | 3V3 (right channel) |

The B-EVN has no user-accessible GPIO button — SW1 is the reset button and SW2 controls programming mode. We therefore wire an external momentary push button between **Header C pin 23B** (FPGA pin 23) and GND. No pull-up resistor is needed on the breadboard because the PCF file uses the `-pullup yes` directive to enable the iCE40's internal pull-up (~100 kΩ):

```
set_io -pullup yes btn  23   # label 23B — Header C
```

This means pressing the button pulls the pin to GND (logic 0), which is the active-low convention our debouncer expects.

### UART and Data Rate

The FTDI chip on the B-EVN board provides a USB-to-UART bridge, so the same USB cable used to program the FPGA also carries the audio data to the laptop. We use 921600 baud.

At 15625 stereo samples per second, each consisting of 4 bytes, the raw data rate is:

```
15625 samples/s × 4 bytes/sample × 10 UART bits/byte = 625,000 bits/s
```

This is well within the 921600 baud capacity, leaving plenty of headroom so no samples are dropped.

## Verilog Implementation

The design is split into four small modules, each responsible for one job. Keeping modules small and focused makes them easier to simulate and reuse.

### Module 1 — I2S Clock Generator (`i2s_clk_gen.v`)

The iCE40UP5K has an internal high-frequency oscillator (HFOSC) running at 48 MHz. We use it as our system clock so that no external crystal is needed. The I2S clock generator divides this down to produce SCK and WS.

Listing 1.1 shows the module declaration.

```verilog
// from i2s_clk_gen.v
module i2s_clk_gen #(
    parameter CLK_DIV        = 24,   // half-period divider for SCK
    parameter BITS_PER_FRAME = 32    // SCK cycles per WS half-period
)(
    input  wire clk,
    input  wire resetn,
    output reg  sck,
    output reg  ws
);
```
*Listing 1.1 — i2s_clk_gen module declaration*

With `CLK_DIV = 24`, the counter toggles SCK every 24 system clocks, giving:

```
SCK = 48 MHz / (24 × 2) = 1 MHz
```

With `BITS_PER_FRAME = 32`, WS toggles every 32 SCK rising edges:

```
WS = 1 MHz / 64 = 15625 Hz  (sample rate)
```

Listing 1.2 shows the counter logic.

```verilog
// from i2s_clk_gen.v
always @(posedge clk) begin
    if (!resetn) begin
        clk_cnt <= 0;
        bit_cnt <= 0;
        sck     <= 0;
        ws      <= 0;
    end else begin
        if (clk_cnt == CLK_DIV - 1) begin
            clk_cnt <= 0;
            sck     <= ~sck;
            // On rising edge of SCK (sck was 0, about to become 1)
            if (!sck) begin
                if (bit_cnt == BITS_PER_FRAME - 1) begin
                    bit_cnt <= 0;
                    ws      <= ~ws;
                end else begin
                    bit_cnt <= bit_cnt + 1;
                end
            end
        end else begin
            clk_cnt <= clk_cnt + 1;
        end
    end
end
```
*Listing 1.2 — SCK and WS generation*

The `clk_cnt` register counts system clock cycles up to `CLK_DIV - 1`, then toggles SCK. We detect the rising edge of SCK by noting that `sck` is currently `0` (it is about to be flipped to `1`). On each such rising edge we increment `bit_cnt` and toggle WS when it wraps.

<div style="border-left: 3px solid #333; padding: 10px 0 10px 20px; margin: 20px 0; font-size: 1em; line-height: 1.6;">
<strong>NOTE:</strong> In an <code>always @(posedge clk)</code> block, all assignments are <em>registered</em> — they take effect at the next clock edge, not immediately. So when we write <code>sck <= ~sck</code> and then check <code>if (!sck)</code>, we are checking the <em>old</em> value of <code>sck</code> before the flip. This is exactly what we want for edge detection.
</div>

### Module 2 — I2S Receiver (`i2s_receiver.v`)

This module watches the SCK and WS signals generated by `i2s_clk_gen` and shifts in the serial data from the microphone, one bit per SCK rising edge.

Listing 2.1 shows the key data path.

```verilog
// from i2s_receiver.v
// Edge detection
wire sck_rise = ( sck && !sck_prev);
wire ws_edge  = (ws  != ws_prev);

// On each SCK rising edge, shift in one data bit
if (sck_rise) begin
    // bit_cnt == 0 is the one-cycle delay slot (I2S spec)
    // Capture DATA_WIDTH bits starting from bit_cnt == 1
    if (bit_cnt >= 1 && bit_cnt <= DATA_WIDTH) begin
        shift_reg <= {shift_reg[DATA_WIDTH-2:0], sd};
    end

    // When all bits are in, latch them to the correct channel
    if (bit_cnt == DATA_WIDTH) begin
        if (ws_latch == 1'b1)
            left_data  <= {shift_reg[DATA_WIDTH-2:0], sd};
        else
            right_data <= {shift_reg[DATA_WIDTH-2:0], sd};
    end
    ...
end
```
*Listing 2.1 — Serial-to-parallel shift register*

The `sck_rise` wire detects a rising edge on SCK by comparing the current SCK value with its registered previous value `sck_prev`. Similarly, `ws_edge` detects any change in WS.

The shift register is `DATA_WIDTH` bits wide (16 bits). On each SCK rising edge where `bit_cnt` is between 1 and 16, we shift in the incoming serial data bit (`sd`) from the right, moving existing bits leftward. This implements MSB-first deserialization. When `bit_cnt` reaches 16, the shift register holds the complete sample and we latch it into either `left_data` or `right_data` depending on `ws_latch` — the value of WS that was recorded when the current frame started.

Listing 2.2 shows how `data_valid` is generated.

```verilog
// from i2s_receiver.v
// data_valid pulses when a complete left→right stereo pair is ready.
// This happens at the WS falling edge (1→0): left channel just finished.
if (ws_edge && ws_prev == 1'b1 && ws == 1'b0) begin
    data_valid <= 1;
end
```
*Listing 2.2 — Stereo sample ready signal*

`data_valid` pulses high for exactly one system clock cycle at the moment WS falls from 1 to 0. At that instant, both `left_data` and `right_data` hold fresh values — the just-finished left channel and the previously latched right channel.

### Module 3 — UART Transmitter (`uart_tx.v`)

The UART transmitter sends a single byte in 8N1 format: one start bit (low), eight data bits LSB first, and one stop bit (high). Listing 3.1 shows the core state.

```verilog
// from uart_tx.v
localparam CLKS_PER_BIT = CLK_FREQ / BAUD_RATE;  // = 52 for 48 MHz / 921600

// Frame: {stop=1, data[7:0], start=0}  — 10 bits total
if (!busy && send) begin
    shift_reg <= {1'b1, data, 1'b0};
    busy      <= 1'b1;
end
```
*Listing 3.1 — UART frame loading*

Once loaded, the shift register drives the TX line one bit at a time. The bit clock is derived from `CLKS_PER_BIT` — the number of 48 MHz system clocks per UART bit period. At 921600 baud this works out to 48,000,000 / 921,600 ≈ 52 clocks per bit. Each bit is held for exactly 52 system clock cycles before the shift register advances.

The `busy` flag prevents the transmitter from being overloaded with new bytes while one is already in flight. The top module checks `busy` before sending each byte.

### Module 4 — Button Debouncer (`debounce.v`)

Mechanical push buttons bounce — they make and break contact many times in the first few milliseconds after a press. Without debouncing, a single press would be seen as many rapid presses. Listing 4.1 shows the debouncing strategy.

```verilog
// from debounce.v
// Two-stage synchroniser eliminates metastability
btn_sync0 <= btn_in;
btn_sync1 <= btn_sync0;

// If input differs from stable output, count clock cycles.
// Only update the output after STABLE_COUNT consecutive agreeing cycles.
if (btn_sync1 != btn_out) begin
    cnt <= cnt + 1;
    if (cnt == STABLE_COUNT - 1) begin
        btn_out <= btn_sync1;
        cnt     <= 0;
        if (btn_sync1 == 1'b0 && btn_out == 1'b1)
            btn_press <= 1'b1;  // one-cycle pulse on falling edge
    end
end else begin
    cnt <= 0;
end
```
*Listing 4.1 — Counter-based debounce*

The input first passes through a two-flip-flop synchroniser chain (`btn_sync0`, `btn_sync1`) to prevent metastability — the condition where a flip-flop's output is neither cleanly high nor low for a period. After synchronisation, a counter counts how long the input has disagreed with the stable output. Only after `STABLE_COUNT` consecutive cycles of disagreement do we accept the new value. At 48 MHz with `STABLE_COUNT = 960,000`, this is a 20 ms debounce window — longer than almost any real button bounce.

### Top Module (`top.v`)

The top module wires all the pieces together and adds the recording state machine. Listing 5.1 shows the use of the iCE40UP5K's built-in HFOSC oscillator primitive.

```verilog
// from top.v
SB_HFOSC #(.CLKHF_DIV("0b00")) hfosc_inst (
    .CLKHFEN(1'b1),
    .CLKHFPU(1'b1),
    .CLKHF(clk)
);
```
*Listing 5.1 — iCE40UP5K internal 48 MHz oscillator instantiation*

`SB_HFOSC` is a Verilog primitive provided by the iCE40 library. `CLKHF_DIV = "0b00"` selects no division, giving the full 48 MHz. `CLKHFEN` and `CLKHFPU` enable the oscillator and its power-up circuitry respectively.

Similarly, the RGB LED is driven via the `SB_RGBA_DRV` primitive, which provides built-in constant-current drive — no external resistors are needed. Listing 5.2 shows the instantiation.

```verilog
// from top.v
SB_RGBA_DRV #(
    .CURRENT_MODE ("0b1"),       // half-current mode
    .RGB0_CURRENT ("0b000001"),  // green ~4 mA
    .RGB1_CURRENT ("0b000001"),  // blue  ~4 mA
    .RGB2_CURRENT ("0b000001")   // red   ~4 mA
) rgba_drv (
    .RGBLEDEN (1'b1),
    .RGB0PWM  (recording),    // green on when recording
    .RGB1PWM  (1'b0),         // blue  always off
    .RGB2PWM  (~recording),   // red   on when idle
    .CURREN   (1'b1),
    .RGB0     (led_green),
    .RGB1     (led_blue),
    .RGB2     (led_red)
);
```
*Listing 5.2 — RGB LED driver via SB_RGBA_DRV primitive*

The recording toggle is a single flip-flop driven by the debounced button press signal.

```verilog
// from top.v
always @(posedge clk) begin
    if (!resetn)
        recording <= 0;
    else if (btn_press)
        recording <= ~recording;
end
```
*Listing 5.3 — Recording toggle*

### UART Byte Sequencer

When `data_valid` pulses and `recording` is active, the top module latches the stereo sample and begins transmitting it 4 bytes at a time. Listing 5.4 shows this sequencer.

```verilog
// from top.v
if (data_valid && recording) begin
    left_latch  <= left_data;
    right_latch <= right_data;
    tx_pending  <= 1;
    tx_byte_idx <= 0;
end

if (tx_pending && !uart_busy && !uart_send) begin
    case (tx_byte_idx)
        2'd0: uart_data <= left_latch[15:8];
        2'd1: uart_data <= left_latch[7:0];
        2'd2: uart_data <= right_latch[15:8];
        2'd3: uart_data <= right_latch[7:0];
    endcase
    uart_send   <= 1;
    tx_byte_idx <= tx_byte_idx + 1;
    if (tx_byte_idx == 2'd3)
        tx_pending <= 0;
end
```
*Listing 5.4 — UART byte sequencer*

The sequencer checks `uart_busy` before asserting `uart_send` to make sure the transmitter is free. At 15625 samples per second and 921600 baud, the UART has ample time to send all 4 bytes before the next sample arrives.

## Simulation

Before testing on real hardware, we simulate the I2S clock generator and receiver together. The testbench in `tb/tb_i2s_receiver.v` models a pair of I2S microphones sending fixed test patterns and verifies that the receiver captures them correctly.

The simulated left mic sends `0xABCD` and the right mic sends `0x1234`. Listing 6.1 shows how the testbench generates the I2S serial data stream.

```verilog
// from tb_i2s_receiver.v
always @(negedge sck) begin
    if (ws != sim_ws_prev) begin
        sim_ws_prev = ws;
        sim_bit_cnt = 0;
        sim_shift   = (ws == 1'b1) ? left_word : right_word;
        sd = 1'b0;  // one-cycle delay slot (I2S spec)
    end else begin
        sim_bit_cnt = sim_bit_cnt + 1;
        if (sim_bit_cnt >= 1 && sim_bit_cnt <= 16) begin
            sd        = sim_shift[15];   // MSB first
            sim_shift = sim_shift << 1;
        end else begin
            sd = 1'b0;
        end
    end
end
```
*Listing 6.1 — Testbench I2S signal generation*

The `negedge sck` trigger is correct because the I2S spec says data is **presented** on the falling edge, and **sampled** on the rising edge by the receiver. Our testbench therefore changes `sd` on falling edges, just as a real microphone would.

## Capturing Audio on the Laptop

Install the only required Python dependency:

```bash
pip install pyserial
```

Find your FPGA's serial port. On Linux it is typically `/dev/ttyUSB0`; on macOS it appears as `/dev/cu.usbserial-XXXX`.

Run the capture script:

```bash
python3 python/capture_audio.py --port /dev/ttyUSB0 --duration 10 --out recording.wav
```

The script will wait for data. Now press the button on the FPGA board — the green LED lights up, the FPGA starts transmitting, and the script displays a progress bar:

```
Opening /dev/ttyUSB0 at 921600 baud …
Recording 10.0s — 156250 samples expected.
Press the FPGA button to start transmitting, then Ctrl+C to stop early.

    8234/156250 samples  ( 5.3%)   0.5s elapsed
```

When the recording is complete (or you press Ctrl+C), the WAV file is written:

```
Captured 156250 stereo samples.
Saved: recording.wav  (610.4 KB)
  Sample rate : 15625 Hz
  Channels    : 2 (stereo)
  Bit depth   : 16-bit signed
  Duration    : 10.00s
```

Press the button again on the FPGA to stop transmission (the green LED turns off and the red LED comes on).

## How the Python Script Works

The script opens the serial port and reads raw bytes in 256-byte chunks. It maintains a buffer and extracts complete 4-byte frames whenever at least 4 bytes are available.

```python
# from capture_audio.py
frame        = buf[:BYTES_PER_FRAME]
left_sample  = bytes_to_signed16(frame[0], frame[1])
right_sample = bytes_to_signed16(frame[2], frame[3])
```

The FPGA sends 16-bit values big-endian (MSB first), but WAV files store samples little-endian. The conversion happens at write time:

```python
# from capture_audio.py
frame_bytes += struct.pack("<hh", l, r)  # little-endian signed shorts
```

The Python `wave` module handles all the WAV header construction automatically.

## Summary

In this project, we implemented the I2S audio protocol from scratch in Verilog on a Lattice iCE40UP5K FPGA. We built a clock generator, a serial-to-parallel receiver, a UART transmitter, and a button debouncer — all small, focused modules that compose cleanly into a working system. A Python script on the laptop reassembles the byte stream into stereo audio and saves it as a standard WAV file.

The most important concept to take away is the relationship between the three I2S signals: SCK drives the bit timing, WS selects the channel, and SD carries the data — one bit per clock, MSB first, with a one-cycle delay after each WS transition. Everything else in this project flows directly from that protocol.

## Going Further

Here are some ideas to extend this project:

- **Increase sample rate** — try 48 kHz by adjusting `CLK_DIV` and `BITS_PER_FRAME` with the HFOSC at 48 MHz. You'll need a higher baud rate or a second serial port.
- **Add a FIFO** — use iCE40 block RAM as a small audio buffer to smooth out any jitter between the I2S sample rate and UART transmission rate.
- **FFT on-chip** — implement a simple DFT in Verilog to display frequency magnitude on LEDs.
- **SPI flash playback** — store a short audio clip in the FPGA's SPI flash and play it back through an I2S DAC.
- **USB Audio** — explore open-source USB Audio class implementations for the iCE40 to stream audio without UART framing overhead.
